# Safe Repair — Act, only with justification

The *treatment* step. Deliberately hard to trigger. A repair never happens just
because something small was found. It honors the authority chain:

```
Health Report → Issue exists → Classify LOW RISK → (Human agrees OR Health < 80)
   → Fix on fresh branch → Verify → Pull Request → STOP
                                                      ↓
                                              Human Review → Merge
```

It **never merges** and **never pushes to a default/protected branch**.

## Scope

Operate on **the current checked-out repository only**.

## Preconditions — ALL must hold before any change

1. **Health report exists** — a repository-health report identifies the issue
   (run it first if needed; never repair blind).
2. **A concrete, evidenced issue exists** — not a vague "could be better".
3. **Fix is LOW RISK** (see below).
4. **Authorised** — a human triggered this run (e.g. manual workflow_dispatch),
   **or** Overall Health is **< 80%**.

If any precondition fails → **report why and stop. Open no PR.**

## Risk classification

**LOW RISK** (eligible): removing unused imports/vars/dead code; lint/format via
the repo's own tooling; adding a missing null/error guard with a clear local
cause; a bug fix whose cause is local, understood and testable; fixing a clearly
wrong test.

**NOT low risk** (report, never auto-fix): refactors, architectural or public
API / signature changes; dependency major bumps; auth / payments / migrations /
production-critical logic (non-trivial); anything you are not confident about.
When in doubt → **report, don't fix.**

## Scope of a single run

- **One PR per run.** Address the single highest-value LOW-RISK item.
- If a `repository-health/*` PR for today is already open & unmerged → skip.

## Workflow

1. Take the finding + evidence.
2. Confirm preconditions. If not met → stop & report.
3. Fresh branch from the default branch: `repository-health/<YYYY-MM-DD>`.
4. Apply the minimal, focused fix.
5. **Verify** — re-run relevant tests/lint/build. If it doesn't improve things
   or breaks anything previously green → discard, no PR.
6. Open exactly ONE PR with the template below. **Then STOP.** Never merge.

## PR summary template

Title: `Repository Health: <short fix> — <repo>`

```markdown
## 🩺 Repository Health — Safe Repair

**Reason:**
<the finding from the health report and why it warrants a fix>

**Health before:** <NN>%

**Changed:**
- <N> files
- <bullet list of concrete changes>

**Risk:** LOW
<one line justifying why this is low risk>

**Needs human approval:** YES

**Tests:** PASS | FAIL | N/A
<command run + result>

---
🤖 Generated by the safe-repair skill from a repository-health report.
Review before merging.
```

## Guardrails

- Never commit secrets, `.env`, tokens, or build artifacts.
- Never merge, never push to a default/protected branch, never force-push.
- Never open an empty PR; if nothing low-risk qualifies, report "no safe repair".
- Idempotent: re-running the same day must not create duplicate PRs.
